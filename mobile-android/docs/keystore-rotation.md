# AndroidKeyStore Key Rotation Runbook

Task #84 introduced two crypto secrets persisted on-device:

1. **`workforce_encryption_key`** — AES-256-GCM key generated and stored
   inside the AndroidKeyStore by `EncryptionService`. Used to wrap:
     - every per-field ciphertext stored in `attendance_submissions`
       (workforceId, GPS lat/lng, timestamps, encrypted-photo-path
       indirection),
     - the SQLCipher passphrase blob stored in
       `db_key_state/wrapped_passphrase_v1` SharedPreferences,
     - the encrypted-photo `*.jpg.enc` files written by `EncryptionService.encryptFile`.

2. **`_androidx_security_master_key_`** — AES-256-GCM master key managed by
   `androidx.security.crypto`'s EncryptedSharedPreferences, used to wrap
   the auth cookie inside `SessionManager`.

Both keys are non-exportable and bound to the device's TEE / StrongBox on
hardware that supports it. They cannot be backed up off-device and are
destroyed by:

- a factory reset,
- an `adb shell pm clear com.luxurycarts.workforce`,
- the user clearing app storage from Settings → Apps,
- an OS upgrade that invalidates the keystore (rare; happens on
  StrongBox-equipped devices when the bootloader is unlocked).

When any of those events occurs, our own data wrapped by the key becomes
unrecoverable. The app MUST gracefully recover by treating the loss as
"new install" — see the recovery section below.

## When to rotate

In normal operation: **never**. The keys are bound to the device, not to
the user's session. Logging out does NOT rotate them; that would force a
re-encryption of every pending submission for no security gain.

Rotate ONLY when:

| Trigger                                                | Action          |
| ------------------------------------------------------ | --------------- |
| Confirmed compromise of a worker device                | Revoke + wipe   |
| `androidx.security.crypto` major version bump          | Re-key & migrate |
| SQLCipher passphrase format change (`wrapped_passphrase_v2`) | Migrate          |
| Suspected keystore weakness disclosed by Google        | Rotate          |

## Rotation procedure (per device)

There are two distinct rotation flows. Choose the right one based on the
trigger.

### Flow A — Compromised device (data MUST NOT survive)

When the device or its keystore is suspected of being compromised, the
goal is to make sure nothing previously protected by the old key is ever
readable again. We deliberately destroy data:

1. Operator suspends the worker account server-side (Task #43 termination).
2. Worker reinstalls the app. `adb shell pm clear` is equivalent if
   physical access is available.
3. Cold start regenerates `workforce_encryption_key` (first
   `getOrCreateKey` call) and `_androidx_security_master_key_` (first
   `MasterKey.Builder.build()` call). A new SQLCipher passphrase is
   generated by `DatabaseKeyManager.getOrCreatePassphrase`.
4. Worker logs in. Server issues a fresh session cookie.

Pending un-synced submissions are intentionally lost. This is correct
behaviour for the compromise threat model.

### Flow B — Proactive / scheduled rotation (data MUST survive)

When the trigger is a library upgrade, a scheduled hygiene rotation, or
a wrapped-passphrase format bump (`wrapped_passphrase_v2`), the user
must not lose pending offline submissions or history. This is the
"in-place re-key" path.

The procedure runs entirely on the device, single-process, on first
cold start after the app upgrade. It is implemented in code as a one-shot
extension of `DatabaseKeyManager` and `EncryptionService`:

1. **Snapshot** — read the current wrapped values:
   - SQLCipher passphrase (`db_key_state/wrapped_passphrase_v1`),
   - column-level master key (`workforce_encryption_key`),
   - per-photo encrypted file headers (still bound to the column key).
2. **Mint the replacement keys** with new aliases (NOT the old ones):
   - `workforce_encryption_key_v2` in AndroidKeyStore,
   - new 32-byte SQLCipher passphrase wrapped under that v2 key,
   - persist as `db_key_state/wrapped_passphrase_v2`.
3. **Atomic re-encrypt of the database** using SQLCipher's
   `PRAGMA rekey` or `sqlcipher_export` against a side file:
     a. Open the live encrypted DB with the old passphrase.
     b. Run `PRAGMA rekey = "x'<new-hex-passphrase>'";` inside a single
        transaction. SQLCipher rewrites every page atomically — either
        the whole DB is re-keyed or the file is left untouched.
     c. Verify by closing, reopening with the new passphrase, and
        running `PRAGMA cipher_integrity_check;`.
3b. **Atomic re-encrypt of column ciphertext and photo files** by
    iterating every row in `attendance_submissions` and every
    `*.jpg.enc` file: decrypt with the old `workforce_encryption_key`,
    re-encrypt with `workforce_encryption_key_v2`, then `WriteAheadLog`
    the new ciphertext back into the same row / replace the file via
    rename-into-place (write to `*.jpg.enc.tmp`, fsync, rename).
4. **Dual-read window** — for one full release cycle the app must accept
   reads with EITHER the old or the new key, so a worker who interrupts
   step 3b mid-flight (battery dies, app killed) can still resume on the
   next cold start. Implement by trying v2 first and falling back to v1
   on `BadPaddingException` / SQLCipher `not a database` errors. The
   rotation worker remains idempotent and re-runs until every row /
   file is on v2.
5. **Atomic alias cutover** — once `db_key_state/rotation_progress` shows
   100% of rows + files migrated AND the DB integrity check passes,
   atomically:
     a. Delete the old `workforce_encryption_key` from AndroidKeyStore.
     b. Delete `wrapped_passphrase_v1` from SharedPreferences.
     c. Bump `db_key_state/active_key_version = 2`.
   These three writes happen in a single SharedPreferences `commit()`
   followed by `KeyStore.deleteEntry`. If the keystore deletion fails,
   the cutover is rolled back by re-flipping `active_key_version` to 1.
6. **Rollback criteria** — abort and revert to v1 if ANY of:
     - `cipher_integrity_check` reports non-OK,
     - more than 1% of rows fail re-encryption with the new key,
     - the AndroidKeyStore returns `KeyPermanentlyInvalidatedException`
       on the v2 alias before cutover (TEE rejected the new key).
   Rollback is "delete v2 alias + delete `wrapped_passphrase_v2` +
   leave `active_key_version = 1`". The next cold start retries from
   step 1.

#### Done criteria for Flow B

- Every pending un-synced submission is still readable after the
  rotation completes.
- History thumbnails decrypt successfully on the first cold start
  after rotation.
- A factory-reset audit shows the v1 alias no longer exists in
  `keystore list` and `wrapped_passphrase_v1` is absent from prefs.
- The release notes attached to the Play upload reference the
  rotation worker version and the commit SHA that introduced it.

NOTE: The current codebase ships only Flow A. Flow B above is the
specification the next bump (e.g. SQLCipher 5.x, or the move to
`wrapped_passphrase_v2`) MUST implement. Tracking lives in this runbook
so the next implementer has the no-data-loss design ready to code
against.

## Recovery from key loss (no compromise)

If a user reports "the app crashes on launch" or "I can no longer see my
history" after an OS upgrade or factory reset:

1. Have them force-stop and reopen the app.
2. If the crash persists, the keystore key is gone but the encrypted
   blobs remain. The app will throw a `SecurityException` from
   `EncryptionService.decrypt` on first read. Today this surfaces as
   `NeedsAttention` rows tagged `DECRYPTION_FAILED` for any pending
   submission, which is intentionally loud.
3. Recovery: clear app data (`Settings → Apps → Workforce → Storage →
   Clear`). The app reinitialises clean. Pending un-synced submissions
   are unrecoverable; synced submissions are server-side and survive.

## Migration smoke test

After bumping any of:

- `net.zetetic:sqlcipher-android` major version,
- `androidx.security:security-crypto` minor version,
- the `KEY_WRAPPED` constant in `DatabaseKeyManager`,

run this manual smoke test on a test device that already has the previous
build installed with at least one synced submission:

1. `adb install -r app-release.apk` (the new build).
2. Cold start the app.
3. Verify in `adb logcat -s DBEncMigration AttendanceSync`:
     - `encrypted DB verified rows=N` (with N matching pre-upgrade count),
     - no `DECRYPTION_FAILED` events on the History screen,
     - no `db_open` events at version 0 (would indicate the schema was
       wiped by destructive fallback).
4. Capture and sync a fresh submission to confirm round-trip still works.

Document the test result and the exact commit SHA in the release notes
attached to the Play upload.
