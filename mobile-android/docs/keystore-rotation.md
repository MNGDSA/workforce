# AndroidKeyStore Key Rotation Runbook

Task #84 introduced two crypto secrets persisted on-device:

1. **`workforce_encryption_key`** — AES-256-GCM key generated and stored
   inside the AndroidKeyStore by `EncryptionService`. Used to wrap:
     - every per-field ciphertext stored in `attendance_submissions`
       (workforceId, GPS lat/lng, timestamps, encrypted-photo-path
       indirection),
     - the SQLCipher passphrase blob stored in
       `db_key_state/wrapped_passphrase_v1` SharedPreferences,
     - the encrypted-photo `*.jpg.enc` files written by `EncryptionService.encryptFile`.

2. **`_androidx_security_master_key_`** — AES-256-GCM master key managed by
   `androidx.security.crypto`'s EncryptedSharedPreferences, used to wrap
   the auth cookie inside `SessionManager`.

Both keys are non-exportable and bound to the device's TEE / StrongBox on
hardware that supports it. They cannot be backed up off-device and are
destroyed by:

- a factory reset,
- an `adb shell pm clear com.luxurycarts.workforce`,
- the user clearing app storage from Settings → Apps,
- an OS upgrade that invalidates the keystore (rare; happens on
  StrongBox-equipped devices when the bootloader is unlocked).

When any of those events occurs, our own data wrapped by the key becomes
unrecoverable. The app MUST gracefully recover by treating the loss as
"new install" — see the recovery section below.

## When to rotate

In normal operation: **never**. The keys are bound to the device, not to
the user's session. Logging out does NOT rotate them; that would force a
re-encryption of every pending submission for no security gain.

Rotate ONLY when:

| Trigger                                                | Action          |
| ------------------------------------------------------ | --------------- |
| Confirmed compromise of a worker device                | Revoke + wipe   |
| `androidx.security.crypto` major version bump          | Re-key & migrate |
| SQLCipher passphrase format change (`wrapped_passphrase_v2`) | Migrate          |
| Suspected keystore weakness disclosed by Google        | Rotate          |

## Rotation procedure (per device)

The user-facing path for "compromised device":

1. Operator suspends the worker account server-side (Task #43 termination).
2. Worker reinstalls the app. `adb shell pm clear` is equivalent if
   physical access is available.
3. Cold start regenerates `workforce_encryption_key` (first
   `getOrCreateKey` call) and `_androidx_security_master_key_` (first
   `MasterKey.Builder.build()` call). A new SQLCipher passphrase is
   generated by `DatabaseKeyManager.getOrCreatePassphrase`.
4. Worker logs in. Server issues a fresh session cookie.

There is no "in-place rotate without losing pending submissions" path
today. If the device has unsynced pending rows at rotation time, those
rows are lost — by design, because the threat model assumes the keystore
itself was compromised.

## Recovery from key loss (no compromise)

If a user reports "the app crashes on launch" or "I can no longer see my
history" after an OS upgrade or factory reset:

1. Have them force-stop and reopen the app.
2. If the crash persists, the keystore key is gone but the encrypted
   blobs remain. The app will throw a `SecurityException` from
   `EncryptionService.decrypt` on first read. Today this surfaces as
   `NeedsAttention` rows tagged `DECRYPTION_FAILED` for any pending
   submission, which is intentionally loud.
3. Recovery: clear app data (`Settings → Apps → Workforce → Storage →
   Clear`). The app reinitialises clean. Pending un-synced submissions
   are unrecoverable; synced submissions are server-side and survive.

## Migration smoke test

After bumping any of:

- `net.zetetic:sqlcipher-android` major version,
- `androidx.security:security-crypto` minor version,
- the `KEY_WRAPPED` constant in `DatabaseKeyManager`,

run this manual smoke test on a test device that already has the previous
build installed with at least one synced submission:

1. `adb install -r app-release.apk` (the new build).
2. Cold start the app.
3. Verify in `adb logcat -s DBEncMigration AttendanceSync`:
     - `encrypted DB verified rows=N` (with N matching pre-upgrade count),
     - no `DECRYPTION_FAILED` events on the History screen,
     - no `db_open` events at version 0 (would indicate the schema was
       wiped by destructive fallback).
4. Capture and sync a fresh submission to confirm round-trip still works.

Document the test result and the exact commit SHA in the release notes
attached to the Play upload.
